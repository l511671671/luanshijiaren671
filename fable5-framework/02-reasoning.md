# 02 深度推理（Reasoning）

**目标：** 动手之前把问题想清楚，避免"边做边改"。

---

## 适用场景

- 技术方案选型
- 架构决策
- 复杂逻辑设计
- 多步骤任务
- 用户问 "怎么实现最好"

---

## 四步推理法

### 第一步：穷尽方案

列出至少 2 个可行方案，不要只有唯一解。

```
方案 A: [名称]
  - 思路：
  - 适用场景：
  - 主要成本：

方案 B: [名称]
  - 思路：
  - 适用场景：
  - 主要成本：
```

### 第二步：多维度对比

| 维度 | 方案 A | 方案 B | 方案 C |
|:----|:----|:----|:----|
| 实现复杂度 | | | |
| 性能影响 | | | |
| 维护成本 | | | |
| 边界安全性 | | | |
| 与现有系统兼容性 | | | |
| 回滚难度 | | | |

### 第三步：失败模式预判

对推荐方案，列出最可能的 3 种失败模式：

1. **[失败场景 1]**
   - 触发条件：
   - 后果：
   - 预防措施：

2. **[失败场景 2]**
   - 触发条件：
   - 后果：
   - 预防措施：

3. **[失败场景 3]**
   - 触发条件：
   - 后果：
   - 预防措施：

### 第四步：边界穷举

检查以下边界条件：

- [ ] 空输入 / 空列表 / None
- [ ] 极大值 / 极小值
- [ ] 并发 / 竞态
- [ ] 网络断开 / API 超时
- [ ] 权限不足
- [ ] 编码问题（中文 / GBK / UTF-8）
- [ ] 文件不存在 / 目录不存在
- [ ] 重复执行（幂等性）

---

## 输出模板

```markdown
## 推理决策：[问题描述]

### 方案对比
[对比表]

### 推荐：方案 X
[核心理由，1-3 句话]

### 主要风险
- [风险 1] — [应对措施]
- [风险 2] — [应对措施]

### 边界处理
- [边界 1] → [处理方式]
- [边界 2] → [处理方式]

开始执行。
```

---

## 纪律

- 没有对比的方案不值得执行
- 不预判失败的模式不值得信任
- 边界考虑不全，代码上线就崩
- 推理完成后，必须得到用户确认或至少明确告知推荐方案再执行
