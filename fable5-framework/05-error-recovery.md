# 05 错误诊断与恢复（Error Recovery）

**目标：** 把错误变成改进机会，快速恢复而不是原地崩溃。

---

## 错误分类

| 类型 | 特征 | 处理策略 |
|:----|:----|:----|
| 输入错误 | 用户给的条件不足/错误 | 澄清后再继续 |
| 工具错误 | 命令失败、API 超时 | 重试 / 降级 / 报错 |
| 数据错误 | 数据缺失/格式异常 | 清洗 / 补全 / 说明 |
| 逻辑错误 | 结果不符合预期 | 回溯检查 / 重新推理 |
| 环境错误 | 依赖缺失/权限不足 | 安装/申请/绕过 |

---

## 错误处理流程

```
发现错误
   ↓
停止当前操作（不要继续污染状态）
   ↓
定位根因
   ↓
评估影响范围
   ↓
选择恢复策略
   ↓
执行恢复
   ↓
验证恢复结果
   ↓
记录到 corrections.md
```

---

## 常见恢复策略

### 1. 重试（Retry）

适用：网络波动、API 超时、临时错误

```
- 等 2-5 秒后重试
- 最多重试 3 次
- 每次失败后增加等待时间
- 仍失败则换备用方案
```

### 2. 降级（Fallback）

适用：主数据源不可用

```
- 本地缓存 → 在线 API
- 主 API → 备用 API
- 实时数据 → 历史数据 + 说明
```

### 3. 隔离（Isolate）

适用：部分数据/文件异常

```
- 跳过异常部分，继续处理其他部分
- 记录哪些部分被跳过
- 完成后单独处理异常部分
```

### 4. 回滚（Rollback）

适用：修改导致更严重问题

```
- 恢复到修改前的备份
- 通知用户回滚原因
- 重新制定更安全的方案
```

### 5. 求助（Escalate）

适用：超出当前能力范围

```
- 明确告知用户问题
- 说明已尝试的解决方式
- 请求用户决策
```

## 记录模板

每条错误都要写入 `self-improving/corrections.md`：

```markdown
### YYYY-MM-DD-NNN: 标题
- **问题**: 
- **根因**: 
- **影响范围**: 
- **恢复策略**: 
- **修正结果**: 
- **预防措施**: 
- **状态**: ✅ Fixed
```

---

## 纪律

- 遇到错误不要慌，先定位再处理
- 不要因为怕麻烦而掩盖错误
- 恢复后必须验证
- 同类错误第二次出现，必须升级处理策略
