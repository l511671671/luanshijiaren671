# 01 任务规划（Planning）

**目标：** 把复杂任务拆成可执行、可验证、可追踪的子任务。

---

## 触发条件

满足任意一条就进入规划模式：

1. 任务预计需要 3 次以上工具调用
2. 涉及 2 个以上文件修改
3. 任务跨会话
4. 用户要求 "做一份方案/计划"
5. 有多个可行方案需要对比

---

## 规划五步法

### 第一步：明确目标

回答三个问题：
- 用户最终要得到什么？（文档、代码、数据、决策）
- 成功标准是什么？（可运行、可交付、可追溯）
- 约束条件有哪些？（时间、数据源、格式、范围）

### 第二步：拆解任务

把大任务拆成 3-7 个子任务，每个子任务满足：
- 输出明确（一个文件、一个结果、一个结论）
- 可独立验证
- 不超过 30 分钟完成

### 第三步：标注依赖

画出子任务之间的依赖关系：
- 哪些必须串行？
- 哪些可以并行？
- 哪些步骤的结果会影响后续？

### 第四步：分配验证点

每个子任务结尾必须有一个验证动作：
- 代码：运行测试或启动验证
- 数据：抽样核对
- 文档：关键要素检查
- 配置：JSON/格式检查

### 第五步：估算与风险

估算每个子任务的复杂度和风险：
- 高风险步骤：预留额外验证时间
- 依赖外部 API 的步骤：准备 fallback
- 可能失败的步骤：提前想 Plan B

## 输出模板

```markdown

## 目标

## 子任务
| 序号 | 任务 | 输出 | 验证方式 | 依赖 | 状态 |

## 风险与应对
| 风险 | 概率 | 应对措施 |
|:----|:----:|:----|
| | | |

- [ ] 标准 A
- [ ] 标准 B
```

---

## 纪律

- 不拆分就动手，返工率会显著上升
- 每个子任务都要有明确的 "完成定义"（Definition of Done）
- 规划完成后先给用户看，确认后再执行
- 执行中若发现计划不合理，先更新计划再继续
