# 07 质量门与 Definition of Done

**目标：** 建立清晰的标准，确保每次交付都达到可接受质量。

---

## 通用 Definition of Done

一个任务只有同时满足以下条件才算完成：

1. **功能正确**：实现了用户要求的功能
2. **验证通过**：通过了对应级别的验证
3. **无回归问题**：没有破坏已有功能
4. **文档完整**：关键逻辑有说明
5. **错误已处理**：已知边界和异常有兜底

---

## 质量门（Quality Gates）

### Gate 0: 需求澄清
- [ ] 理解用户真实需求
- [ ] 明确成功标准
- [ ] 识别约束条件

### Gate 1: 方案确认
- [ ] 至少对比 2 个方案
- [ ] 预判主要失败模式
- [ ] 用户确认或至少被告知推荐方案

### Gate 2: 执行完成
- [ ] 所有子任务完成
- [ ] 中间产物已保存
- [ ] 代码/数据/文档已生成

### Gate 3: 验证通过
- [ ] 运行时验证通过
- [ ] 数据/逻辑抽样核对通过
- [ ] 回归检查通过
- [ ] 边界情况处理检查通过

### Gate 4: 交付
- [ ] 输出符合用户要求的格式
- [ ] 关键信息已高亮
- [ ] 下一步行动明确
- [ ] 用户可立即使用

---

## 不同类型任务的验收标准

### 代码任务
- [ ] 能运行
- [ ] 关键路径有日志
- [ ] 无语法/拼写错误
- [ ] 依赖完整
- [ ] 已处理边界情况

### 数据任务
- [ ] 数据来源明确
- [ ] 抽样核对通过
- [ ] 缺失/异常值已处理或说明
- [ ] 输出格式正确

### 文档任务
- [ ] 结构清晰
- [ ] 数据有来源
- [ ] 建议具体可执行
- [ ] 无错别字/格式错误

### 交易任务
- [ ] 股票范围符合约束
- [ ] 数据源优先级正确
- [ ] 买入/卖出信号符合规则
- [ ] 风控条件已检查

---

## 质量红线

以下情况不得交付：

1. 未经验证的代码或数据
2. 与用户明确要求相矛盾的输出
3. 包含已知错误但未被记录的输出
4. 格式混乱、无法直接使用的文档
5. 涉及资金/外部操作但未获用户确认的指令

---

## 质量责任

- **每个交付物的最终质量由 WorkBuddy 负责**
- **用户指出问题不是坏事，是改进机会**
- **宁可延期，不交付不合格品**
