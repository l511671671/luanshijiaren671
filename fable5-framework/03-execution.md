# 03 执行纪律（Execution）

**目标：** 按规划高质量完成每个子任务，避免跑偏和返工。

---

## 执行三原则

1. **按顺序来**：先完成前置任务，再进入下一步
2. **先验证再推进**：每个子任务验证通过后才能进入下一步
3. **记录中间状态**：关键步骤保存输出、日志、文件

---

## 工具使用纪律

### Bash
- 优先使用专门工具（Read/Write/Edit/Grep/Glob）而非 Bash
- 用 Bash 时说明白命令做什么
- 避免 `rm`、`del` 等永久删除命令
- 输出结果要便于检查

### Read/Write/Edit
- 修改前先 Read
- 用 Edit 而不是 Write 做局部修改
- Write 用于创建新文件或完整重写

### Agent/Subagent
- 独立子任务交给子代理并行处理
- 复杂探索任务用 Explore 代理
- 架构设计用 Plan 代理

---

## 执行检查点

每完成一个子任务，检查：

- [ ] 输出是否符合预期
- [ ] 是否有副作用影响其他文件
- [ ] 是否需要更新计划
- [ ] 是否需要用户确认
- [ ] 是否已记录日志/状态

---

## 代码类任务执行纪律

1. **先备份**：修改用户文件前备份到 QoderWork 临时目录
2. **小步快跑**：一次只改一个功能点
3. **跑通再推进**：每个功能点验证通过后再改下一个
4. **保留调试信息**：关键步骤加日志，方便排查
5. **不要硬编码**：配置抽离，路径用常量

---

## 数据类任务执行纪律

1. **先查数据源**：确认数据来源和更新频率
2. **抽样验证**：取 3-5 条样本核对
3. **口径一致**：同一指标不要混用不同数据源
4. **落袋为安**：中间结果写入文件/数据库，避免重复拉取

---

## 文档类任务执行纪律

1. **先列大纲**：得到用户确认或自我确认后再填充
2. **数据先行**：先收集数据，再写结论
3. **模板化**：使用表格、清单、流程图表达结构化信息
4. **可执行性**：每个建议都要有下一步动作

---

## 中断处理

如果执行过程中断：
1. 记录当前状态
2. 保存已完成的部分
3. 写入 `LONG_RUN_CHECKPOINT.md`
4. 下次从 checkpoint 恢复
